# Анализ масштабируемости: Метод простой итерации

## Наблюдения

### Область эффективного масштабирования (p = 1–16)
- Ускорение **близко к линейному** до p ≈ 16
- Version 2 показывает стабильно лучшее ускорение, чем Version 1
- При p = 16: 
  - Version 1 ≈ 11.2×
  - Version 2 ≈ 12.5× (эффективность ~78%)

### Область насыщения (p > 16)
- После 16 потоков рост ускорения сильно замедляется
- При p = 32: Version 2 достигает максимума ≈ 17.0×
- При p = 40 ускорение практически не растёт (Version 1 — 15.5×, Version 2 — 16.8×)
- Version 2 опережает Version 1 на 10–25% на большинстве значений p

## Причины ограниченной масштабируемости

1. **Memory-bound характер задачи** — матрица 25 000 × 25 000 (~5 ГБ) не помещается в кэш
2. **Конкуренция за память** — все потоки интенсивно читают матрицу A
3. **NUMA-эффект** — система имеет 2 NUMA-узла, при p > 20 возрастает межсокетный трафик
4. **Накладные расходы** — барьеры и проверка сходимости (особенно заметно в Version 1)

## Вывод

**Оптимальное число потоков: 16–24**

- Version 2 предпочтительнее — меньше overhead'ов и лучше масштабируется
- До p = 16 ускорение растёт хорошо
- После p = 24 прирост минимальный, эффективность падает ниже 50%
- Использование 40 потоков **нецелесообразно** — ускорение почти не увеличивается
